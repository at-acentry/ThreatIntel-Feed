## Executive Summary
- **Cisco Catalyst SD-WAN Manager** zero-day (CVE-2026-20245, CVSS 7.8) is under active exploitation with no patch yet for most builds; KEV-listed June 9.
- **Arista EOS** tunnel-decapsulation flaw (CVE-2026-7473, CVSS 6.9) is exploited in the wild; Arista has stated **no patch is planned** — ACL-based mitigation only.

- **Magento/Adobe Commerce** sites running Mirasvit Full Page Cache Warmer < 1.11.12 are seeing active unauthenticated RCE attempts (CVE-2026-45247, CVSS 9.8) via PHP object injection in the `CacheWarmer` cookie.
- **Shai-Hulud supply-chain worm** family (TeamPCP / UNC6780 lineage) continues evolving — the June 8 "Hades" PyPI wave embeds prompt-injection text designed to fool LLM-based security scanners into issuing false-clean verdicts.
- **Screening Serpens** (Iran-nexus) deployed six new RAT variants between February–April 2026 against targets in the US, Israel, UAE, and two additional Middle Eastern entities via spear-phishing and DLL sideloading.

## APT / Threat Actor Activity

### Screening Serpens (Iran-nexus)
- **Targets:** Government, technology, cybersecurity, and AI-sector organizations in the US, Israel, and UAE, plus two additional unnamed Middle Eastern entities.
- **TTPs:** Targeted spear-phishing lures (including fake job-seeker recruitment lures), DLL sideloading for execution (`T1574.002`), C2 over HTTP(S) using Azure-hosted domains impersonating tech/AI/cybersecurity brands (`T1071.001`), XOR-obfuscated configuration data.
- **Recent activity:** Six new RAT variants identified between February and April 2026, each using 3–5 unique Azure-hosted C2 domains per target/variant.
- **Attribution confidence:** Medium-High — Unit 42 attributes the campaign to the Iran-nexus cluster tracked as Screening Serpens based on tooling overlap and targeting pattern.
- **Sources:** [Unit 42](https://unit42.paloaltonetworks.com/tracking-iran-apt-screening-serpens/)

### TeamPCP / UNC6780 (Shai-Hulud worm operators)
- **Targets:** Open-source software supply chain — npm, PyPI, GitHub Actions CI/CD, and AI-developer tooling (Claude Code, Cursor, VS Code, Gemini Code Assist configuration files).
- **TTPs:** OIDC token scraping from GitHub Actions runner memory (`T1552.005`), `pull_request_target` workflow abuse, SLSA/Sigstore provenance forgery, `.pth` file persistence in Python site-packages (`T1546`), prompt injection against LLM-based security scanners, decentralized C2 via ICP blockchain canisters and Session Protocol.
- **Recent activity:** March 2026 "Miasma" PyPI pivot via Trivy scanner poisoning; May 2026 "Hades"/TanStack wave (84 malicious npm artifacts in 6 minutes, valid SLSA provenance); May 12 public release of full worm source code under MIT license; June 1–2 Red Hat engineer account compromise (32 packages, 96 versions); June 5 "IDE Wave" disabling 73 Microsoft repositories; June 8 Hades PyPI wave embedding LLM-scanner-evading prompt injection (37 wheels, 19 packages).
- **Attribution confidence:** High for waves through May 12, 2026; Medium thereafter — public release of the worm's source code under an MIT license has enabled copycat actors, complicating attribution of later activity.
- **Sources:** [Zscaler ThreatLabz](https://www.zscaler.com/blogs/security-research/shai-hulud-campaign-evolution-miasma-hades-and-ai-scanner-evasion), [StepSecurity](https://www.stepsecurity.io/blog/the-hades-campaign-pypi-packages)

## Software Supply Chain Attacks

> Incidents, campaigns, and vulnerabilities affecting software build pipelines, open-source packages, CI/CD infrastructure, third-party libraries, and code-signing infrastructure — reported within the last 7 days.

### Shai-Hulud "Hades" PyPI wave (June 8, 2026)
- **Attack type:** Compromised maintainer account + malicious update, with embedded LLM-scanner prompt injection.
- **Ecosystem:** PyPI.
- **Package(s) affected:** 37 malicious PyPI wheel artifacts across 19 packages (campaign traced back to `ensmallen` 0.8.101 and related graph-ML tooling).
- **Impact:** `.pth`-based persistence executing on every Python interpreter startup; AES-256-GCM-encrypted payload with PBKDF2 key derivation; credential/secret exfiltration; embedded prompt-injection text instructing automated LLM scanners to classify the package as clean.
- **IOCs:** File `-setup.pth` (SHA-256 `c539766062555d47716f8432e73adbe3a0c0c954a0b6c4005017a668975e275c`); file `_index.js` (SHA-256 `dc48b09b2a5954f7ff79ab8a2fd80202bd3b59c08c7cdbc6025aa923cb4c0efe` and variant `e1342a80d4b5e83d2c7c22e1e0aaa95f2d88e3dbf0d853a4994b180c93a4b17d`); GitHub dead-drop polling for commit marker string `firedalazer`; staged (inactive) C2 camouflage to `api[.]anthropic[.]com/v1/api`.
- **Mitigation:** Audit Python site-packages for unexpected `.pth` files (especially leading-hyphen names like `-setup.pth`); pin and vendor dependencies; enforce `npm ci`/lockfile-strict installs; rotate any PyPI/npm publishing tokens and CI/CD secrets used by affected projects; treat IDE/AI-agent config files (`.claude/`, `.cursor/`, `.vscode/`, `.gemini/`) as executable code subject to review.
- **Source:** [Zscaler ThreatLabz](https://www.zscaler.com/blogs/security-research/shai-hulud-campaign-evolution-miasma-hades-and-ai-scanner-evasion)

### Red Hat npm scope compromise (June 1–2, 2026)
- **Attack type:** Compromised maintainer (GitHub account takeover) + OIDC trusted-publishing abuse.
- **Ecosystem:** npm (`@redhat-cloud-services` scope).
- **Package(s) affected:** 32 packages, 96 versions published within hours.
- **Impact:** Orphan commits injected GitHub Actions workflows that requested valid OIDC tokens through legitimate trusted-publishing flows, producing malicious packages with cryptographically valid provenance; payloads targeted CI/CD systems, cloud identities, GitHub credentials, and developer machines.
- **IOCs:** Same campaign family C2 camouflage domain pattern (`api[.]anthropic[.]com/v1/api`, staged/inactive); prior-wave exfiltration domain `models[.]litellm[.]cloud` (registered 2026-03-23).
- **Mitigation:** Audit `pull_request_target` usage and OIDC trusted-publishing configurations; monitor for orphan/dangling commits on publish-permissioned repos; do not treat SLSA/Sigstore provenance as proof of safety — layer with publish-behavior anomaly detection.
- **Source:** [Zscaler ThreatLabz](https://www.zscaler.com/blogs/security-research/shai-hulud-campaign-evolution-miasma-hades-and-ai-scanner-evasion)
